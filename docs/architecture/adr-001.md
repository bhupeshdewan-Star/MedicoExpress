# ADR-001: PostgreSQL Exclusive Architecture

## Context
During the pilot readiness phase of ClinCommand OS™, database connections supported a dual SQLite3 and PostgreSQL driver configuration. SQLite served as a fallback for local offline development, while PostgreSQL was designed as the production layer. In a regulated GxP environment, maintaining two distinct database engines introduces validation risk, data synchronization overhead, and potential vulnerabilities from unvalidated offline paths.

## Decision
Purge the SQLite3 database connection driver and remove all SQLite bootstrap operations and file-based data stores. PostgreSQL is established as the exclusive, mandatory database engine for all execution phases (development, testing, validation, and production). If a valid PostgreSQL database pool connection cannot be verified on server startup, the bootstrap sequence must log a critical failure and terminate immediately.

## Alternatives Considered
* **Keep SQLite for Local Development:** Rejected because SQLite lacks native Row-Level Security (RLS) constraints, custom triggers, and schema verification tools required for enterprise GxP validation. It causes disparity between staging and local environments.

## Risks
* **Local Test Overhead:** Running tests locally now requires an active PostgreSQL environment (or a mocked interface).
* **Mitigation:** Fallback logic is added to test scripts to bypass active network connections using secure mock models during dry-run executions.

## Compliance Impact
Ensures that all development stages run on the identical schema, indexes, and triggers as production, satisfying FDA 21 CFR Part 11 and EU Annex 11 validation expectations.

## Rollback Strategy
Roll back to a dual-mode branch, restore sqlite3 dependencies, and re-enable the conditional SQLite3 configuration file in `db.js`.
