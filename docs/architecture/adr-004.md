# ADR-004: Merkle Audit Vault Architecture

## Context
Regulated compliance rules (e.g. FDA 21 CFR Part 11) dictate that system audit trails must be secure and immutable. While standard database triggers can prevent deletion, a database administrator (superuser) could theoretically bypass triggers, modify records, and cover their tracks.

## Decision
Establish a cryptographic Merkle Audit Vault. All database transactions write audit log rows. Periodically (or on-demand), unsealed logs are grouped into "blocks" and hashed using SHA-256 in a Merkle tree structure. The resulting Merkle root, block hash, and previous block hash are stored in the database. This links audit blocks together in a tamper-evident chain. A validation route audits this chain and reports the first compromised node if hashes mismatch.

## Alternatives Considered
* **Immutable Database (e.g. AWS QLDB):** Considered but rejected to avoid vendor lock-in and keep the platform database-agnostic within a standard PostgreSQL architecture.
* **Basic Row Hashing:** Rejected because verifying a basic hash list is slow and does not support fast, hierarchical integrity checks.

## Risks
* **Write Latency:** Re-computing the Merkle chain on every single write degrades API throughput.
* **Mitigation:** Run the sealing utility asynchronously or trigger it on validation cycles, separating high-speed writes from validation checks.

## Compliance Impact
Establishes a verifiable, non-repudiation audit vault satisfying GxP requirements for computer systems validation (CSV) and electronic records.

## Rollback Strategy
Revert to simple audit log triggers (UPDATE/DELETE prevention) and disable the Merkle verify API route.
