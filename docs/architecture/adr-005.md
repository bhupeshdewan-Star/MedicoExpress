# ADR-005: AI Governance Architecture

## Context
Deploying Generative AI within clinical and regulatory workflows requires strict validation. Undocumented prompt changes can lead to hallucinations or quality drift in generated summaries and patient safety narratives, which violates GxP compliance.

## Decision
Implement an AI Governance Center. All AI prompts are registered in a database repository (`prompt_registry`) and version-controlled. Any prompt changes trigger a formal approval workflow (`Draft → Review → Approved → Retired`). The system enforces that only "Approved" prompt versions are executed in workspaces. We also log model costs, API latencies, and output verification checks (PII/PHI detection, toxicity) to ensure safety and auditability.

## Alternatives Considered
* **Hardcoded Prompt Templates:** Rejected because it makes prompt updates require code deployments and validation cycles.
* **Dynamic User Prompts (No Governance):** Rejected as it completely violates GxP validation rules.

## Risks
* **LLM Vendor Drift:** External model endpoints (e.g. OpenAI/Gemini) can change their underlying weights, causing outputs to drift even if the prompt remains static.
* **Mitigation:** Enforce daily validation tests that run sample inputs against the prompts and check that output semantics remain stable.

## Compliance Impact
Aligns with WHO, EMA, and FDA AI/ML regulatory frameworks for software quality management and risk governance.

## Rollback Strategy
Disable registry checks and allow local prompt templates fallback inside `ragService.js`.
