# ADR-006: SSO & SCIM Architecture

## Context
Enterprise clients expect unified identity lifecycle management. Users must authenticate through their existing identity providers (IdPs like Azure AD, Okta, Google Workspace), and employee status changes (provisioning/de-provisioning) must propagate automatically to ClinCommand OS™.

## Decision
Design a dual Identity Integration Layer:
1. **SSO Core:** Support SAML 2.0 and OIDC authentication mappings, storing IdP certificates and routing rules inside `sso_configurations`.
2. **SCIM Provisions:** Expose standard SCIM 2.0 endpoints at `/api/scim/v2/Users` and `/api/scim/v2/Groups` supporting GET, POST, PUT, and PATCH operations. This allows automatic synchronization of user directories, de-provisioning, and role syncing.

## Alternatives Considered
* **SSO-only (No SCIM):** Rejected because SSO only provisions users during login ("just-in-time"), which fails to instantly deactivate employees upon termination.

## Risks
* **Incorrect Role Mapping:** Synced groups from IdPs must map correctly to application permission levels.
* **Mitigation:** Implement strict mappings matching IdP group names to the internal `roles` list.

## Compliance Impact
Guarantees fast user de-provisioning within minutes of corporate separation, conforming to SOC2 and ISO 27001 Access Control requirements.

## Rollback Strategy
Disable SCIM route validation and fallback exclusively to manual user creation and local username/password login configurations.
