# ADR-002: Multi-Tenant RLS Model

## Context
As a multi-tenant life sciences platform, ClinCommand OS™ must isolate clinical, medical affairs, and regulatory data across distinct client organizations (tenants). Application-level isolation (e.g. adding `WHERE tenant_id = X` in backend queries) is prone to developer oversight, potentially leading to critical cross-tenant data leaks.

## Decision
Enforce Row-Level Security (RLS) at the database layer on all business-critical tables. All API request threads propagate tenant IDs through an `AsyncLocalStorage` context to the PostgreSQL client. Queries are executed within transaction blocks that run `SET LOCAL app.current_tenant_id = X`. This restricts table visibility to rows matching the active tenant ID, bypassing database access if the parameter is empty or incorrect.

## Alternatives Considered
* **Database-per-Tenant Separation:** Considered but rejected due to high operational cost, resource overhead, and complexity of running migration schemas across hundreds of database instances.
* **Schema-per-Tenant Separation:** Rejected because running custom migrations on separate schemas complicates GxP software validations.

## Risks
* **RLS Bypass during Bootstrapping:** System startup scripts and user login queries must run under a superuser context (or empty tenant setting) to fetch users and boot system states.
* **Mitigation:** Database policies explicitly allow access if `current_setting('app.current_tenant_id')` is empty, which is restricted to internal bootstrap and system authentication stages.

## Compliance Impact
Provides ironclad data isolation that complies with HIPAA, SOC2 Trust Services Criteria, and FDA 21 CFR Part 11 database controls.

## Rollback Strategy
Disable database RLS on all tables: `ALTER TABLE <name> DISABLE ROW LEVEL SECURITY` and drop the custom row policies.
