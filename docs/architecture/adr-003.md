# ADR-003: Refresh Token Security Model

## Context
Regulated users require continuous session access during long appraisal workflows without encountering arbitrary sign-out disruptions. Standard JWT access tokens are short-lived for security, but refresh mechanisms must be secured against credential theft and session hijacking.

## Decision
Implement JWT access tokens with a short lifespan (1 hour) paired with rotating refresh tokens (7 days). Refresh tokens are double-hashed (SHA-256) before database commits and tracked inside the `refresh_tokens` table under a common `token_family` uuid. Refreshing a token revokes the old token and inserts a new one in the same family. If a previously rotated (revoked) token is presented (indicating a replay attack), the system revokes the entire token family tree and logs a high-severity security event.

## Alternatives Considered
* **Stateless JWTs only:** Rejected because stateless tokens cannot be revoked before their expiration time if a compromise occurs.
* **Persistent Sessions (Cookies Only):** Rejected to allow flexible integration across native mobile and web frontends.

## Risks
* **Token Bloat in Database:** Continual rotation generates a large volume of revoked rows in the database.
* **Mitigation:** Implement a nightly cron script to purge expired and revoked refresh tokens older than 14 days.

## Compliance Impact
Satisfies FDA 21 CFR Part 11 Session Timeout and Electronic Signature Authentication criteria, ensuring that compromised tokens are immediately neutralized.

## Rollback Strategy
Disable token rotation in `auth.js` and fall back to stateless access tokens or static refresh token lifespans.
